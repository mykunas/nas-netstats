# NAS NetStats 开发规则

本文件是 NAS NetStats 后续开发的强制规则。每次开发前必须先阅读 `README.md` 和本文件。

## 项目定位

NAS NetStats 是一个轻量级 NAS 实时网速与历史流量统计工具，通过 Docker 部署，网页端查看数据。

项目只关注 NAS 本机可采集网卡的整体收发流量，不分析应用、不分析容器、不分析设备明细。

## 必须实现的范围

1. 查看 NAS 当前下载速度。
2. 查看 NAS 当前上传速度。
3. 统计今日下载流量。
4. 统计今日上传流量。
5. 统计今日总流量。
6. 统计本月下载流量。
7. 统计本月上传流量。
8. 统计本月总流量。
9. 支持历史流量统计。
10. 支持日视图、周视图、月视图、年视图。
11. 支持网页端查看具体数据。
12. 支持 Docker Compose 一键部署。

## 明确不做

以下功能不允许主动加入，除非用户后续明确修改项目范围：

1. 不做 Docker 容器流量统计。
2. 不做 qBittorrent、Transmission、Aria2 等下载器统计。
3. 不做代理流量统计。
4. 不做 OpenWrt 多设备统计。
5. 不做复杂告警。
6. 不做 AI 分析。
7. 不做多 NAS 节点管理。
8. 不做用户权限系统。

## 技术栈

- backend: FastAPI
- frontend: React + Vite + TypeScript
- chart: ECharts
- database: PostgreSQL
- collector: Python
- deploy: Docker Compose

## 数据采集规则

1. collector 默认汇总采集 NAS 本机所有可采集网卡的整体流量，也允许通过环境变量限制为单个网卡。
2. 数据来源只能使用 `/proc/net/dev` 或 `/host/proc/net/dev`。
3. 不允许读取 Docker、下载器、代理、路由器、多设备或应用级统计接口。
4. 网卡采集范围必须通过环境变量配置，默认 `NAS_INTERFACE=all`。
5. 采集间隔必须通过环境变量配置。
6. collector 只记录累计字节数与采样时间，复杂业务逻辑优先放在 backend。

## 配置规则

1. 所有可变配置必须放在环境变量或 `.env.example` 中。
2. 不允许把密码、Token、连接串硬编码到代码里。
3. Docker Compose 使用 `.env` 或默认环境变量启动。
4. 新增配置时必须同步更新 `.env.example` 和 `README.md`。

## 前端规则

1. 页面目标是清晰、实用、像传统流量统计工具。
2. 优先展示当前速率、今日流量、本月流量、历史图表和历史明细。
3. 不做营销首页。
4. 不主动加入用户登录、权限、主题市场、AI 解读、复杂告警等高级功能。
5. 图表库使用 ECharts。

## 前端 UI 重构规则

项目现在的前端目标：NAS NetStats 要做成一个单页科技感仪表盘，用户打开一个页面就能看到 NAS 实时网速、今日流量、本月流量、历史统计、提醒信息。

### 布局规则

1. 桌面端采用左右布局。
2. 左侧为主信息区，占页面 65% - 75% 宽度。
3. 右侧为历史统计区，占页面 25% - 35% 宽度。
4. 平板端可以上下排列。
5. 手机端必须单列显示。
6. 页面不能因为窗口缩放而错位。
7. 所有卡片、图表、表格需要自适应容器大小。

### 视觉规则

1. 整体风格：科技感、扁平化、干净、现代。
2. 使用卡片式设计。
3. 避免花哨渐变过多。
4. 深色模式下要有高级感。
5. 浅色模式下要清爽。
6. 数据数字要突出。
7. 图表要简洁。
8. 表格要紧凑但清晰。
9. 颜色要有层次，不要杂乱。
10. 保留历史统计条形图和日历热力图。

### 禁止事项

1. 不要添加 Docker 容器流量统计。
2. 不要添加下载器流量统计。
3. 不要添加代理流量统计。
4. 不要把多个页面做得太复杂。
5. 不要引入过重 UI 框架。
6. 不要破坏现有 API。
7. 不要移除已有核心功能。

### 完成标准

1. `npm run build` 通过。
2. 首页单页仪表盘可以正常显示。
3. 实时刷新正常。
4. 历史列表和历史图表分开。
5. 深色模式正常。
6. 移动端、平板端、桌面端都能基本适配。

## 后端规则

1. 后端只提供 NAS 网卡流量相关 API。
2. API 应保持简单，优先满足网页展示与 Docker 部署。
3. 统计逻辑基于采样点之间的累计字节差值。
4. 遇到网卡计数器重置时，只统计正向增量，避免产生负流量。

## Docker 规则

1. Docker 部署必须简单。
2. 优先保证 `docker compose up -d --build` 可以启动。
3. Compose 中至少包含 PostgreSQL、backend、collector、frontend。
4. collector 容器需要只读挂载宿主机 `/proc` 到 `/host/proc`。

## 验证规则

每个模块完成后需要说明如何验证：

1. backend: 说明健康检查和 API 验证方式。
2. frontend: 说明网页访问地址和主要页面验证点。
3. collector: 说明如何确认采样数据写入数据库。
4. deploy: 说明 Docker Compose 启动与日志检查方式。

## 开发原则

1. 优先保证项目可以运行。
2. 保持实现轻量，避免主动扩大范围。
3. 保持目录结构清晰。
4. 修改已有代码前先理解现有实现。
5. 不为尚未要求的高级功能预留复杂架构。
